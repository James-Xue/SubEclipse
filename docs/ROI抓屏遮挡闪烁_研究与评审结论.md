# ROI抓屏遮挡闪烁：研究与评审结论

版本: v1.0  
日期: 2026-03-13

## 1. 问题定义
你描述的问题成立：

1. 当前实现是先抓屏、再检测、再在 overlay 画黑框。
2. 下一帧抓屏会把已画出的黑框再次抓进去。
3. 检测器在被黑框覆盖后可能检测不到文字，主线程就把 `mask_boxes` 清空。
4. 再下一帧又抓到文字，黑框又出现，形成消失-出现的周期闪烁。

这不是简单参数没调好，而是典型的渲染-采集闭环反馈问题。

## 2. 代码证据

1. 抓屏来源是根窗口合成结果：`src/platform/x11/x11_capture.cpp` 中 `XShmGetImage`/`XGetImage` 都对 `impl_->root` 抓取。
2. 主线程每轮用最新检测结果整包替换遮挡框：`src/main.cpp` 中 `mask_boxes = std::move(latest_boxes);`。
3. overlay 每次先清空再重画：`src/platform/x11/overlay_window.cpp` 的 `OverlayWindow::draw()` 先 `XClearWindow`，再按 `mask_boxes` 填充。
4. 检测为空时，主线程会把 `mask_boxes` 变为空，直接导致该帧黑框消失。

## 3. 对“窗口句柄逐窗采集+合成再识别”的可行性评估

结论：理论上可研究，工程上不推荐作为当前阶段主路径。

### 3.1 能做的部分（X11）
1. 可枚举窗口树并做 ROI 相交筛选（例如 `XQueryTree` + 属性查询）。
2. 可尝试通过 Composite 扩展获取窗口像素源做本地合成。

### 3.2 难点与风险
1. 合成一致性难：窗口透明、阴影、变换、特效、重定向后纹理等很难和真实桌面 1:1 对齐。
2. Z-order 与遮挡关系复杂：动态窗口变化频繁，维护成本高。
3. 性能成本高：逐窗抓取+CPU/GPU合成会显著增加延迟和占用。
4. 兼容性差：不同 WM/Compositor 差异大；Wayland 下模型完全不同。
5. 维护成本高：方案复杂度远高于当前项目阶段，回归风险大。

## 4. 子代理结论评审（3轮）

### 第1轮评审：事实正确性
1. 通过 root 抓屏导致 overlay 污染检测输入，这个判断正确。
2. 闪烁时序链条（有框 -> 无框 -> 有框）判断正确。

结论：核心根因判断通过。

### 第2轮评审：方案可行性
1. 子代理提出“逐窗口句柄合成”高复杂度、低性价比，这个方向判断基本正确。
2. 其给出的“在抓屏线程直接 hide/show overlay”建议存在高风险：
   - 当前架构下 `OverlayWindow` 在主线程使用，抓屏在线程中；直接跨线程操作 X11 窗口容易引入竞态。
   - 该方式本身也可能引入可见闪烁和时序抖动。

结论：不建议直接采用该实现路径。

### 第3轮评审：落地策略与项目匹配度
1. 子代理文档总体结构完整，但部分改造清单过于激进，不适合直接开做。
2. 更符合当前仓库的路线应是“状态稳定化优先、架构大改后置”。

结论：文档需修订后再作为实施基线。本文件即为修订版。

## 5. 推荐实施路径（按优先级）

### P0: 先做抗闪烁状态机（最小改动，优先）
1. 不要在单帧检测为空时立即清空 `mask_boxes`。
2. 引入 TTL/迟滞（hysteresis）：
   - 连续 N 帧未命中才移除某个框（例如 N=2~4）。
   - 新框命中连续 M 帧后再确认显示（例如 M=1~2）。
3. 可选：给框做短暂最小显示时长（例如 120~200ms）。

收益：通常能显著降低闪烁；改动小、风险低。

### P1: 做检测与渲染去抖
1. 对检测框做 IoU 关联与平滑（已有 pipeline 可扩展）。
2. 把“当前检测结果”改成“稳定跟踪结果”再交给主线程绘制。

收益：减少闪烁和跳框，提升观感。

### P2: 再评估采集隔离方案
1. 若 P0/P1 后仍不满足验收，再评估更重方案。
2. “逐窗口句柄合成”仅建议作为探索分支，不作为主线里程碑。

## 6. 对你的问题的直接结论

1. 这类问题本质是底层采集-渲染闭环的架构问题，不是简单打断点就能解决。
2. 你提出的“按句柄抓窗口再合成识别”在理论上可研究，但当前项目阶段不建议走主线。
3. 推荐先做抗闪烁状态机和结果稳定化，这通常是同类实时遮挡系统的首选工程解。
